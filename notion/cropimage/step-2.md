## 현재 구현된 전체 흐름

현재 앱의 사용 흐름은 아래처럼 이어집니다.

1. 이미지 파일을 열거나 클립보드 이미지를 붙여넣는다
2. 저장 폴더를 확인하거나 변경한다
3. 화면에서 여러 개의 크롭 사각형을 만든다
4. 사각형 위치를 조정하거나 삭제한다
5. `설정` 버튼으로 현재 상태를 확정한다
6. `분할 시작`으로 각 영역을 개별 파일로 저장한다
7. 필요하면 JSON 설정 파일로 저장해 다음 작업에 다시 불러온다

이 흐름이 중요한 이유는, 사용자가 지금 어디까지 진행했는지 UI만 봐도 이해할 수 있기 때문입니다. 저장이 가능한 조건도 분명하고, 작업 단계가 한눈에 보입니다.

## 1. 이미지 불러오기와 클립보드 붙여넣기

앱의 시작점은 두 가지입니다.

- 로컬 이미지 파일 열기
- `Ctrl + V`로 클립보드 이미지 붙여넣기

여기서 가장 큰 차별점은 클립보드 지원입니다. 실제 작업에서는 파일 탐색기에서 이미지를 찾기보다, 캡처 후 바로 붙여넣는 경우가 훨씬 많기 때문입니다. 이 기능 하나만으로도 작업 진입 속도가 꽤 달라집니다.

현재는 아래 상황을 처리할 수 있습니다.

- 클립보드에 실제 이미지 데이터가 있으면 바로 로드
- 클립보드에 파일 목록이 있으면 지원 형식 이미지 중 첫 번째 파일 로드
- 지원하지 않는 데이터면 경고 메시지 표시

덕분에 "캡처 → 붙여넣기 → 사각형 지정 → 저장"이라는 짧은 흐름이 자연스럽게 연결됩니다.

## 2. 크롭 사각형 생성, 이동, 삭제

이 프로젝트의 핵심 UI는 결국 캔버스 위에서 사각형을 다루는 부분입니다. 이미지를 띄워놓고 원하는 영역을 직접 지정할 수 있어야 실제 작업 속도가 나옵니다.

현재 가능한 동작은 다음과 같습니다.

- 빈 영역을 드래그해서 새 사각형 만들기
- 기존 사각형을 드래그해서 위치 이동
- `Delete` 키 또는 우클릭으로 선택한 사각형 삭제
- `전체 초기화`로 모든 사각형 제거
- `Esc`로 현재 생성 중인 사각형 취소

여기서 중요한 포인트는, 사각형을 마음대로 수정할 수는 있지만 저장 직전에는 반드시 `설정` 버튼으로 현재 상태를 확정한다는 점입니다. 이 구조 덕분에 저장 순서를 관리하기 쉬워지고, 편집 중인 상태가 그대로 저장되는 실수도 줄일 수 있습니다.

## 3. 확대, 축소, 맞춤 보기 기능을 넣은 이유

이미지 분할 프로그램에서는 생각보다 배율 조정이 중요합니다. 특히 세로로 긴 이미지나 해상도가 큰 이미지는 화면에 한 번에 다 보이지 않기 때문에, 확대와 축소가 없으면 정확한 위치를 잡기 어렵습니다.

현재 지원하는 보기 조정 기능은 아래와 같습니다.

- `확대`
- `축소`
- `100%`
- `맞춤`
- `Ctrl + 마우스휠` 확대/축소
- `Shift + 마우스휠` 가로 스크롤

이 기능들을 넣으면서, 큰 이미지를 빠르게 훑고 필요한 부분만 확대해서 세밀하게 사각형을 맞출 수 있게 됐습니다.

## 4. JSON 설정 저장과 불러오기가 중요한 이유

이번 프로젝트에서 개인적으로 만족도가 높았던 부분 중 하나가 바로 JSON 설정 저장 기능입니다. 같은 형식의 이미지를 반복해서 잘라야 하는 경우, 매번 좌표를 처음부터 다시 잡는 것은 꽤 비효율적이기 때문입니다.

현재 JSON 파일에는 아래 정보가 저장됩니다.

- 원본 이미지 경로
- 이미지 입력 방식
- 저장 폴더 경로
- 원본 이미지 크기
- 현재 줌 값
- 설정 확정 여부
- 크롭 사각형 좌표 목록

불러오기에서는 아래 상황도 함께 처리합니다.

- 저장된 원본 이미지 경로가 남아 있으면 자동으로 다시 열기
- 저장 당시 이미지 크기와 현재 이미지 크기가 다르면 좌표를 비율에 맞춰 자동 보정
- 저장된 폴더가 없으면 다시 선택하도록 안내

이 기능이 들어가면서 이 프로젝트는 단순한 일회성 툴이 아니라, 반복 작업을 도와주는 유틸리티에 더 가까워졌습니다.

## 5. 저장 폴더와 결과 파일명 처리

막상 실제로 써보면, 저장 성공 여부만큼 중요한 것이 "어디에 어떤 이름으로 저장됐는지"입니다. 결과물이 정리되지 않으면 후속 작업이 더 번거로워지기 때문입니다.

현재 저장 방식은 아래처럼 동작합니다.

- 사용자가 저장 폴더를 직접 선택할 수 있음
- 파일 기반 입력이면 원본 확장자를 최대한 유지
- 클립보드 기반 입력이면 기본적으로 `.png` 저장
- 파일명은 `원본이름_rect_01` 형식으로 순차 번호 부여
- 같은 이름이 이미 있으면 `_1`, `_2`를 붙여 충돌 방지

이렇게 해두면 저장 결과물이 뒤섞이지 않고, 이후 업로드나 정리 단계에서도 파일 순서를 유지하기 편합니다.

## 6. `분할 시작` 버튼을 조건부로 활성화한 이유

프로그램을 만들다 보면 버튼은 있는데 왜 눌리지 않는지 애매한 경우가 종종 생깁니다. 이번 프로젝트에서는 그런 모호한 상태를 줄이기 위해 `분할 시작` 버튼이 활성화되는 조건을 분명하게 두었습니다.

현재 활성화 조건은 다음과 같습니다.

- 이미지가 열려 있어야 함
- 사각형이 하나 이상 있어야 함
- 최근 수정 이후 `설정`을 다시 눌러 확정돼 있어야 함
- 저장 폴더가 유효해야 함

이 기준을 둔 이유는 단순합니다. 저장 시점의 상태를 예측 가능하게 만들기 위해서입니다. 사용자가 왜 저장이 안 되는지 바로 이해할 수 있어야 프로그램도 덜 답답하게 느껴집니다.

## 7. Windows EXE 빌드까지 고려한 이유

Python 프로젝트는 코드로만 남겨두면 결국 나만 쓰는 도구로 머무르기 쉽습니다. 반대로 EXE 빌드까지 준비해두면 배포와 사용 장벽이 훨씬 낮아집니다. 그래서 현재 구조는 실행 코드뿐 아니라 Windows EXE 빌드까지 함께 고려해 정리해두었습니다.

현재 기준으로는 아래 흐름을 갖추고 있습니다.

- `requirements-build.txt`로 빌드 의존성 관리
- `build_exe.bat`로 빌드 실행
- PyInstaller 기반 단일 실행 파일 생성
- 결과물은 `dist\AutoCropSplitter.exe`

이 단계가 중요한 이유는, 단순한 개인 스크립트에서 실제 사용 가능한 앱으로 넘어가는 기준점이 되기 때문입니다.
